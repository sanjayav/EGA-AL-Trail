# ADR 0004 · Ed25519 + did:web for DPP signing

**Status** Accepted · 2026-05-04

## Context

Every DPP must be cryptographically verifiable per ESPR Article 9 and
SDD §3 Layer 3. We need:

- A signature suite supported by W3C Verifiable Credentials 2.0.
- A DID method that's deployable without external dependencies.
- A canonicalisation strategy that survives JSON serialisation round-trips.

## Decision

- **Signature suite**: `Ed25519Signature2020` (W3C VC 2.0 cryptosuite).
- **Key algorithm**: Ed25519. Deterministic, fast, 64-byte signatures,
  conservative side-channel posture.
- **DID method**: `did:web` for the platform issuer, resolvable at
  `https://dpp.ega.ae/.well-known/did.json` in production, `did:web:dpp.ega.local`
  in dev. External verifiers (DNV, Bureau Veritas, ASI accredited firms)
  use their own DID methods; we only require the resulting credential
  validates against W3C VC 2.0.
- **Canonicalisation**: deterministic JSON encoding (sorted keys, no extra
  whitespace) for v1.0. Full RDF Dataset Canonicalisation (URDNA2015) is
  deferred to v1.5 when the EU DPP Registry publishes its JSON-LD context.
- **Multibase encoding**: base58btc with `z` prefix per W3C VC convention.

## Why not ECDSA / RSA / EdDSA-Ed448

- ECDSA P-256 is fine but more complex to implement safely (deterministic
  vs randomised signatures, side-channel pitfalls in some libraries).
- RSA is too heavy and the keys too large for a system that may sign
  hundreds of thousands of DPPs per year per tenant.
- Ed448 has weaker library support and offers no practical security
  advantage at our threat model.

## Why not full URDNA2015 canonicalisation today

- The canonicalisation algorithm is complex and slow on large payloads.
- The EU DPP Registry context isn't published yet; designing for it now
  would be premature.
- Deterministic JSON works well for a single-issuer model and is trivially
  verifiable. We accept the migration cost in v1.5.

## Consequences

- The dev key is generated on first use and persisted in
  `infra/dev-keys/issuer-ed25519.pem`. **In production this lives in HSM/KMS**;
  the signer abstracts the key access so swapping in AWS KMS or Azure Key
  Vault is a single-file change.
- DID document publication is the platform's responsibility. The first
  production task is to serve `https://dpp.ega.ae/.well-known/did.json` with
  the issuer's public key.
- External verifier credentials (DNV CFP VCs, ASI CoC VCs) are stored as
  references in the DPP body, not embedded. We verify them at issuance time
  and on every authority-tier read.
